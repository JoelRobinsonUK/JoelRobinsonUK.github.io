Hackspace Portfolio:
Floriofeel

Initial Concept

Floriofeel is a mood tracking app that uses imagery of flowers based on Victorian era floriology (the language and meanings of flowers). The user would be able to select the flower that represents their current feeling and be shown how many users had also felt that way in the last hour.

This idea was generated in the ideation session for the collaborative project, however it was sidelined as its link to “connection” was tenuous.

Two of the modifiers lend themselves to this concept. “Monochrome” will guide the look and feel of the app interface, while “elevate” led to the idea of the app giving users small activities they might undertake to improve their mood (or make the most of a good mood).


Look and Feel

I knew I wanted the landing page of the app to feature a flowering meadow as its background, so I set about searching for one that seemed appropriate. Having found one (iStock, 2014) I uploaded it to an image to palette generator at hexcolor.co, this website gave me a wide range of colours picked directly from the image. I picked a shade of light purple that I felt would be a good neutral accent for the app that still was still floral in feel.

Image from iStock

Once I had this core colour selected I put it into the Adobe Color palette tool and looked through the options it gave. The monochromatic palette ended up including a good range of shades that will give the app a consistent look while still having variety.

The palette options from Adobe Color

I only settled on the name Floriofeel as I began to design the main logo of the app. The original logo used the Ebrima font, and a flower icon taken from Flaticon (Freepik, 2020).

The monochrome Floriofeel logo

The logo with a splash of colour

I intended to use the monochrome version as the yellow was both hard to see on the landing page and contrasted too much with the accent colour of the navigation bar. However based on feedback from Nishesh, who has plenty of UX experience, I use the logo with yellow colouring, as it was suggested that version was easier to read for the most part.

Later in the development process when I settled on the flowers I would be using to present emotions, I found that the pack of icons I used for the prototype (Freepik, 2020) had most of the flowers I needed, and the few it didn’t I could slightly edit to meet my needs.

The finalised flower icons

Sample of the Quicksand font

In order to give the app more of its own visual identity I searched for a custom font from the google fonts library. I settled on “Quicksand”, a sans serif font with rounded lettering that matches the rounded elements of the app interface.

After adding icons and the font I realised that the logo no longer fit with the look of the app. I designed a new logo that incorporated one of the unused icons from the flowers pack. The daisy had a yellow center that reminded me of the existing logo so I used that, and I employed the Quicksand font with a black outline for a cohesive style.

The updated Floriofeel logo


User Journey

The first step in designing this app was figuring out a prototype user journey, where would the user go to access each feature of the app?

User journey ptototype rendered in Adobe Photoshop

I settled on a main line progression that takes the user from the landing page to sign in/up, and from there to the home page. The homepage features the nine flowers/emotions that the user can choose, which I mean to select after some research.

Landing page, sign in, and home page

Once the user has picked their flower they proceed to a screen showing them how many other users felt this way recently, from this page they can choose to read more about the flower or move on to the suggested activities (which I refer to as “nudges”). It may end up being more user friendly to include the nudges on the number display page, provided the list is fairly short.

Number page, nudges page, and user history

From anywhere in the app the user will be able to view their history, this information will be largely for their own purposes and there will be no visualisation of their data. They will also be able to add a note to their previous selections for their own reference, perhaps after they’ve reflected on what caused that feeling or how they got out of it.

sign up, flower information, and history note page


Progressive Web Apps

I had intended to develop this app in Construct, a Javascript based mobile game engine. However on further reflection I decided that the limitations of construct (such having to manually position elements) outweighed the benefit of my being familiar with the software.

To that end I made the decision to build Floriofeel as a Progressive Web App, I have had some experience developing the front end of a PWA before and so was able to quickly generate the interface almost exactly as I designed it in the user journey.

Using PWA technologies allows for easy testing and quick deployment between versions. It also presents an opportunity to learn more about Javascript and web technologies, a branch of development that I haven’t had much experience in previously.

Development

Database and Authentication

I decided to use Google’s Firebase database for the back end of the application, as it seems the easiest to learn of the options available to me. Firebase also includes functionality for user authentication alongside its database, making it the obvious choice for user accounts.

The database contains information about the nine flower options the user has to choose from, which are loaded into the app from the database whenever needed. Having this data stored there instead of locally in the app makes it easier for me to update or replace the information when necessary.

The database structure for flower data.

The database also stores the users’ previous choices, so their history is persistent across devices when they’re logged in. I achieved this by creating a new document in the database whenever a user signs up, with the user’s UID token as the name of the document.

[I am aware from my research while developing this functionality that using the UID is not best practice, however I had issues using Firebase’s included GetToken() function, so for this prototype I simply chose to use something I knew would work.]

Whenever the user selects a flower from the homepage of the app it sends an update to the “history” collection saved in their document, with all the key information about the flower, as well as the time and date that they logged their feelings. The most important piece of data stored in each entry is a timestamp number with a “-” appended to the front. This allows me to sort by the timestamp when retrieving the user’s history, and have their data displayed newest to oldest.

An example of a user's history in the database.

When the user views their history a card is generated for each entry in their database document. They can open these cards to find a couple of functionality options.

If the user hasn’t written a note yet they will see a text area where they can add a note giving more detail about their feelings at that time. Once they’ve written a note it is pushed to the entry’s document in the database and displayed as normal text in the modal card. Existing notes can be edited using the pencil button.

The user can also delete entries from their history at any time using the red trash can button, this deletes the entry permanently from the database and reloads the history page.
